LangGraph ：State、Node、Edge 的基本概念

之前我们使用langchain的时候，是一条线型的，输入 -》 步骤1 -> 步骤2 -> 步骤3 -> 输出，如果你想要执行步骤2的时候回到步骤1， 进入一个循环那就不行了，因为我们知道Agent的核心是ReAct的模式，需要进入循环模式， 用户输入 -> 思考 -> 调用工具 -> 查看结果 -> 不够 -> 继续调用工具-> ♻️ -> 输出， 但是线形的方式无法表达“往回调“的功能呢，那么我们就需要一个能允许循环的方式，那么这个方式就是图

一、LangGraph 是什么？一句话讲清楚

LangGraph 是 LangChain 团队开源的 有状态工作流编排框架。它把 Agent 的执行过程建模为一张有向图（Directed Graph）：

  • Node（节点） 负责干活——调 LLM、执行工具、跑业务逻辑
  • Edge（边） 负责调度——决定”下一步该走哪个节点”
  • State（状态） 负责通信——所有 Node 共享一份可读可写的状态数据

三者配合的关系很简单：Node 读写 State，Edge 基于 State 决定流转方向。

LangGraph 在 2025 年 10 月发布了稳定的 v1.0 版本（LTS），目前最新版本为 v1.2.x。如果你之前看过一些用 set_entry_point() 的老教程，那些是 v0.x 的 API，已经被废弃了。本文全部基于 v1.0+ 的稳定 API。

二、State：图的”共享内存”

2.1 概念

State 是整张图的中央数据总线。每个 Node 在执行时都会拿到当前的 State 快照作为输入，执行完后返回一份 partial update，由框架负责合并回去。

你可以把 State 理解成一个在所有节点之间流转的”信封”——每个节点打开信封读取内容，做完处理后把新的内容塞回去，再传给下一站。

2.2 定义 State Schema

State 的结构（Schema）支持三种定义方式：

方式一：TypedDict（最常用）

方式二：Pydantic BaseModel（需要数据校验时）

方式三：dataclass

三种方式功能上等价，TypedDict 是社区最广泛采用的方式，因为它足够轻量，也是官方文档的默认选择。

2.3 Channel（通道）：State 中的每个字段

LangGraph 内部把 State 的每个字段称为 Channel。这不是什么高深概念，就是你在 TypedDict 里定义的每个 key。

之所以叫 Channel 而不直接叫 field，是因为每个 Channel 可以单独配置更新策略（Reducer），这是 LangGraph 状态管理最核心的设计。

2.4 Reducer：解决并行写入冲突的关键

想象一个场景：Node A 和 Node B 同时并行运行，两者都想更新 results 字段。如果没有任何额外处理，后写入的就会覆盖先写入的——这就是 Race Condition。

Reducer 函数就是用来告诉 LangGraph：当多个 Node 同时更新同一个字段时，应该怎么合并。

常用的 Reducer 有：

Reducer 用途 示例
operator.add 列表拼接 / 整数累加 [1] + [2] → [1, 2] 或 1 + 2 → 3
operator.concat 字符串拼接 "hello " + "world" → "hello world"
add_messages 消息列表合并（带去重） 自动按 message ID 去重，新消息追加

2.6 State 使用的核心规则

  1. Node 返回 partial update：Node 不需要返回完整的 State，只需要返回需要更新的字段

  2. 没有 Reducer 的字段默认覆盖：所以并行场景下一定要给可能被并发写入的字段加 Reducer

  3. State 是不可变的输入：Node 拿到的 State 是当前快照的只读拷贝，修改它不会影响全局

三、Node：图的”工人”

3.1 概念

Node 就是一个 Python 函数。它接收当前 State，做一些计算或副作用（调 LLM、访问数据库、执行工具），然后返回对 State 的更新。

3.5 几种典型的 Node 类型

从功能角度看，Node 通常分为这几类：

LLM Node——调用大模型

Tool Node——执行工具调用

Logic Node——纯业务逻辑

四、Edge：图的”交通信号灯”

4.1 概念

Edge 定义了节点之间的流转规则。如果说 Node 是”做什么”，那 Edge 就是”做完之后去哪里”。

LangGraph 提供了四种 Edge 机制，按复杂度递增：

4.2 普通 Edge：固定跳转

最简单的情况——A 做完一定去 B。

START 和 END 是 LangGraph 的两个虚拟节点：

  • START：图的入口，invoke() 调用时从这里开始
  • END：图的出口，走到这里整个图就执行完毕

4.3 条件 Edge：动态路由

根据当前 State 的内容，决定下一步去哪个节点。

条件入口：从 START 就开始分支

4.4 Command：路由 + 状态更新一步到位

v1.0 引入的 Command 是一个很实用的原语。传统做法中，如果你想在路由的同时更新 State，需要把更新逻辑放在 Node 里、路由逻辑放在 Edge 的路由函数里——分成两个地方维护。Command 让你在一个 Node 函数里同时完成这两件事：

使用 Command 时，不需要再为这个 Node 注册条件 Edge——路由逻辑已经内嵌在返回值里了。

4.5 Send：动态扇出（Map-Reduce）

当你需要把一个列表中的每个元素分别发送给同一个 Node 并行处理时，Send 就派上用场了。典型场景：生成了 N 个主题，要对每个主题分别调用 LLM 生成内容。

Send 接收两个参数：目标 Node 名称 + 传给该 Node 的 State。每个 Send 会创建一个独立的执行分支。

五、编译与执行：从 Builder 到 Runner

5.1 compile()：把蓝图变成可执行对象

StateGraph 本身只是一个 Builder——你用它来注册 Node 和 Edge。真正可以运行的是 compile() 之后返回的 CompiledStateGraph。

编译时会做一些基础校验：检查有没有孤立节点、Edge 引用的节点是否存在等等。

5.2 invoke()：同步执行

5.3 stream()：流式执行

stream 模式下，每个 Node 执行完都会 yield 一个事件，适合需要实时反馈的场景（比如聊天界面逐步展示思考过程）。

八、常踩的坑

1. 并行节点不加 Reducer 导致数据丢失

如果两个 Node 同时从 START 出发并行执行，都更新同一个字段但没有 Reducer，后执行完的会覆盖前一个的结果。解决方案很简单：给并发写入的字段加 Annotated[list, operator.add]。

2. 条件 Edge 的路由函数返回值和 mapping 对不上

add_conditional_edges 的 mapping 字典里必须包含路由函数所有可能的返回值，漏掉一个就会在运行时报错。建议用 Literal 类型注解来约束返回值。

3. 忘记 compile()

StateGraph 是 Builder，不能直接调用 invoke()。必须先 compile() 得到 CompiledStateGraph 才能执行。

4. Node 里直接修改 state 对象

State 是只读快照，在 Node 里做 state["messages"].append(...) 不会生效。正确做法是通过返回值来更新。

5. 混用普通 Edge 和条件 Edge

同一个源节点不要既加 add_edge 又加 add_conditional_edges，两条路径可能同时触发，产生非预期行为。

九、小结

LangGraph 的设计哲学可以用一句话概括：用图的思维来编排 Agent 的执行流程。

State 提供了节点间通信的统一机制，Reducer 解决了并行写入的冲突问题；Node 是纯粹的计算单元，可以是任意 Python 函数；Edge 把这些计算单元串联成有意义的工作流，支持固定跳转、条件路由、动态扇出等多种模式。

掌握了这三个核心概念，你就掌握了 LangGraph 80% 的能力。剩下的 20%（Checkpointer 持久化、Human-in-the-loop、子图嵌套等）都是在这个基础上的扩展，有了扎实的基础，上手会很快。

建议的学习路径：先动手跑通本文的 ReAct Agent 示例，然后逐步加入 Checkpointer 实现多轮对话，最后尝试用条件 Edge 和 Send 构建更复杂的多 Agent 协作系统。
